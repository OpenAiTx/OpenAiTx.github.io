Installation

Unter macOS oder Linux kannst du The Fuck über [Homebrew][homebrew] installieren:

brew install thefuck

Unter Ubuntu / Mint installiere The Fuck mit folgenden Befehlen: 

sudo apt update
sudo apt install python3-dev python3-pip python3-setuptools
pip3 install thefuck --user

Unter FreeBSD installiere The Fuck mit folgenden Befehlen: 

pkg install thefuck

Unter ChromeOS installiere The Fuck mit chromebrew und folgendem Befehl: 

crew install thefuck

Unter Arch-basierten Systemen installiere The Fuck mit folgendem Befehl: 

sudo pacman -S thefuck

Auf anderen Systemen installiere The Fuck mit pip:

pip install thefuck

Alternativ kannst du einen Paketmanager des Betriebssystems (OS X, Ubuntu, Arch) verwenden.

# Es wird empfohlen, diesen Befehl in deine .bash_profile, .bashrc, .zshrc oder ein anderes Startskript einzutragen:

eval $(thefuck --alias)

Du kannst auch ein beliebiges Alias verwenden, z.B. für Montage:

eval $(thefuck --alias FUCK)

Oder in deiner Shell-Konfiguration (Bash, Zsh, Fish, Powershell, tcsh).

Änderungen sind nur in einer neuen Shell-Sitzung verfügbar. Um Änderungen sofort wirksam zu machen, führe source ~/.bashrc (oder deine Shell-Konfigurationsdatei wie .zshrc) aus.

Um korrigierte Befehle ohne Bestätigung auszuführen, verwende die Option --yeah (oder kurz -y, oder --hard, wenn du besonders frustriert bist):

fuck --yeah

Um Befehle rekursiv zu korrigieren, bis sie erfolgreich sind, verwende die Option -r:

fuck -r

Aktualisierung

pip3 install thefuck --upgrade

Hinweis: Die Alias-Funktionalität wurde in Version 1.34 von The Fuck geändert

Deinstallation

Um The Fuck zu entfernen, führe die Installationsschritte rückwärts aus:

  • Entferne oder kommentiere die Alias-Zeile für thefuck aus deiner Bash-, Zsh-, Fish-, Powershell-, tcsh-, ... Shell-Konfiguration
  • Deinstalliere die Binärdateien mit deinem Paketmanager (brew, pip3, pkg, crew, pip)

Funktionsweise

The Fuck versucht, den vorherigen Befehl mit einer Regel abzugleichen. Wird eine Übereinstimmung gefunden, wird ein neuer Befehl mit der passenden Regel erstellt und ausgeführt. Die folgenden Regeln sind standardmäßig aktiviert:

  • adb_unknown_command – korrigiert falsch geschriebene Befehle wie adb logcta;
  • ag_literal – fügt -Q zu ag hinzu, wenn vorgeschlagen;
  • aws_cli – korrigiert falsch geschriebene Befehle wie aws dynamdb scan;
  • az_cli – korrigiert falsch geschriebene Befehle wie az providers;
  • cargo – führt cargo build statt cargo aus;
  • cargo_no_command – korrigiert falsche Befehle wie cargo buid;
  • cat_dir – ersetzt cat durch ls, wenn versucht wird, ein Verzeichnis mit cat zu öffnen;
  • cd_correction – Rechtschreibprüfung und Korrektur fehlgeschlagener cd-Befehle;
  • cd_cs – ändert cs in cd;
  • cd_mkdir – erstellt Verzeichnisse, bevor mit cd hineingegangen wird;
  • cd_parent – ändert cd.. in cd ..;
  • chmod_x – fügt das Ausführungsbit hinzu;
  • choco_install – hängt gängige Suffixe an chocolatey-Pakete an;
  • composer_not_command – korrigiert composer-Befehlsnamen;
  • conda_mistype – korrigiert conda-Befehle;
  • cp_create_destination – erstellt ein neues Verzeichnis, wenn du versuchst, mit cp oder mv in ein nicht existierendes zu kopieren/verschieben;
  • cp_omitting_directory – fügt -a hinzu, wenn du ein Verzeichnis mit cp kopierst;
  • cpp11 – fügt fehlendes -std=c++11 zu g++ oder clang++ hinzu;
  • dirty_untar – korrigiert tar x, das ins aktuelle Verzeichnis entpackt hat;
  • dirty_unzip – korrigiert unzip, das ins aktuelle Verzeichnis entpackt hat;
  • django_south_ghost – fügt --delete-ghost-migrations zu fehlgeschlagenen Ghosts Django South Migrationen hinzu;
  • django_south_merge – fügt --merge zu inkonsistenten Django South Migrationen hinzu;
  • docker_login – führt ein docker login aus und wiederholt den vorherigen Befehl;
  • docker_not_command – korrigiert falsche docker-Befehle wie docker tags;
  • docker_image_being_used_by_container – entfernt den Container, der das Image verwendet, bevor das Image entfernt wird;
  • dry – korrigiert Wiederholungen wie git git push;
