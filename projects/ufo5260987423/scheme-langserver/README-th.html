Scheme-langserver

>หมายเหตุ: มีบั๊กจำนวนมากใน scheme-langserver ฉันกำลังแก้ไขและขอความช่วยเหลือจากชุมชน กรุณาอดทนรอ

>หมายเหตุ: คุณสามารถดูข้อมูลชนิดที่สร้างอัตโนมัติได้ที่ ที่นี่ ซึ่งปัจจุบันใช้สำหรับการพัฒนาในขั้นถัดไป (อาจรวมถึง AKKU) และการดีบัก

การเพิ่มฟีเจอร์อย่าง autocomplete, ไปยังนิยาม หรือเอกสารขณะ hover เป็นงานที่ต้องใช้ความพยายามมากสำหรับการเขียนโปรแกรม อย่างไรก็ตาม เมื่อเทียบกับภาษาอื่นเช่น java, python, javascript และ c การพัฒนา language server protocol สำหรับภาษา lisp นั้นเหมือนทำในสุญญากาศ Geiser, racket langserver และ swish-lint เป็นต้น งานของพวกเขาทั้งหมดอิงกับ repl (Read-Eval-Print Loop) หรือ keyword tokenizer แทนที่จะเป็นการเขียนโปรแกรมจริง ตัวอย่างเช่น หากโปรแกรมเมอร์กำลังเขียนโค้ดในโปรเจกต์ที่ยังไม่สมบูรณ์ ซึ่งโค้ดยังไม่สามารถรันได้ทั้งหมด Geiser หรือเครื่องมืออื่น ๆ จะเติมเฉพาะ binding identifiers ในระดับ top-level ที่แสดงโดย procedure environment-symbols (สำหรับ Chez) หรือเติม symbol แทน identifier ซึ่งหมายความว่า สำหรับ local binding และโค้ดที่ยังไม่สมบูรณ์ แม้ว่าการวิเคราะห์โค้ดจะมีความสำคัญเป็นอย่างมาก Geiser และเครื่องมืออื่น ๆ กลับไม่ช่วยอะไรในเรื่องการระบุขอบเขตของ identifier ที่ถูกต้อง กรณีเช่นเดียวกันนี้ก็เกิดกับฟีเจอร์ goto definition และฟีเจอร์อื่น ๆ อีกมาก

สาเหตุหลักคือ สำหรับ scheme และ lisp dialects อื่น ๆ ชุดข้อมูลที่หลากหลายและโครงสร้างควบคุมที่ยืดหยุ่น ทำให้การวิเคราะห์โปรแกรมเป็นความท้าทายใหญ่ โดยเฉพาะ macro ดูเหมือนว่า scheme จะถูกใช้สำหรับงานโปรแกรมขั้นสูงและ meta/macro programming เป็นหลัก แต่ฉันขอแย้งว่าไม่ใช่ Scheme สามารถสร้างสิ่งที่น่าสนใจได้มากมายหากมีสภาพแวดล้อมการเขียนโปรแกรมที่ดี และตอนนี้คุณสามารถ เริ่มต้นที่นี่

แพ็กเกจนี้เป็นการติดตั้ง language server protocol ที่ช่วยในการพัฒนาโปรแกรมด้วย scheme โดยให้ฟีเจอร์ completion, definition และ type inference ฟีเจอร์เหล่านี้สร้างจากการวิเคราะห์โค้ดแบบ static โดยอิงตาม มาตรฐาน r6rs และกฎพื้นฐานบางอย่างสำหรับโค้ดที่ยังไม่สมบูรณ์ แพ็กเกจนี้และไลบรารีที่เกี่ยวข้องได้รับการเผยแพร่หรือกำลังจะเผยแพร่ด้วย Akku ซึ่งเป็น package manager สำหรับ Scheme

แพ็กเกจนี้ได้ผ่านการทดสอบกับ Chez Scheme เวอร์ชัน 9.4 และ 9.5 และจะมีการทดสอบโดยละเอียดในเวอร์ชัน 10.0.0 หลังจากอัปเกรดระบบปฏิบัติการแล็ปท็อปของฉัน

ฉันทำงานโอเพ่นซอร์สนี้ในเวลาว่าง และฉันสามารถนำเสนอไอเดียดี ๆ ให้กับชุมชนมากมาย เช่น ฝัง data flow analysis ลงใน scheme-langserver หรืออย่างอื่นอีกมากมาย และฉันกำลังขอรับการบริจาคหรือเงินทุนสนับสนุนเพิ่มเติมอย่างต่อเนื่อง คุณสามารถบริจาคผ่าน patreon นี้ หรือ 爱发电

การคอมไพล์ ติดตั้ง & ตั้งค่า ดูได้ที่ ที่นี่

สถานะล่าสุด

ฉันจะยังคงแก้ไขบั๊ก, โปรไฟล์โค้ด, และรวบรวมข้อมูลสำหรับหนังสือเล่มใหญ่ของฉันเกี่ยวกับระบบ type inference ที่ทำเอง ซึ่งจะใช้เวลาประมาณ 1 ปี การพัฒนาต่อไปจะรวมถึงปลั๊กอิน VScode และการวิเคราะห์ data flow แต่จริง ๆ แล้ว ตอนนี้ฉันถือว่างานโอเพ่นซอร์สนี้เป็นงานพาร์ทไทม์ และไม่สามารถรับประกันตารางเวลาได้

การปล่อยเวอร์ชัน

1.2.9 ตอนนี้ สนุกกับ type inference ได้เลย!

สำหรับเวอร์ชันก่อนหน้า กรุณาดูที่ ไฟล์นี้

ฟีเจอร์

  • เติมคำสั่ง binding ของ identifier ทั้งในระดับ top-level และ local
  • ไปยังนิยาม (Goto definition)
  • ใช้งานร่วมกับ package manager: Akku ได้
  • ซิงโครไนซ์การเปลี่ยนแปลงไฟล์และอัปเดตดัชนีที่เกี่ยวข้อง
  • Hover
  • อ้างอิงและไฮไลท์ในเอกสาร (Document-scoped references)
  • Document symbol
  • จับ binding ของ local identifier ที่ใช้ *-syntax (define-syntax, let-syntax ฯลฯ)
  • ดัชนีแบบขนานข้ามแพลตฟอร์ม
  • ตัวใส่คำอธิบายซอร์สโค้ดที่สร้างเอง เพื่อให้รองรับไฟล์ .sps
  • Peephole optimization สำหรับคำขอ API ด้วยงานที่สามารถพักได้ (suspendable task)
  • Type inference ด้วย DSL interpreter ที่ทำเอง (ฉันภูมิใจมาก!) และตอนนี้ได้ฝังเข้าไปในระบบ auto-completion แล้ว ดังภาพตัวอย่างด้านล่าง "length-b" และ "length-a" ที่มีชนิด "integer?" จะอยู่หน้าสุดของตัวเลือกที่แนะนำ เพราะตรงกับชนิดพารามิเตอร์ที่ต้องการจาก "<="
สามารถทดสอบผลลัพธ์นี้ได้โดยรัน scheme --script tests/protocol/apis/test-completion.sps และไฟล์ log scheme-langserver.log จะมีผลลัพธ์แบบนี้:
