expr_codegen ตัวแปลงนิพจน์

ภูมิหลังของโครงการ

หลังจากที่ผมได้เปิดตัวไลบรารี polars_ta ใหม่ ก็กลับมาทบทวนว่า expr_codegen คืออะไร

expr_codegen โดยแท้จริงคือ DSL หรือภาษาเฉพาะทาง (Domain Specific Language) แต่ไม่ได้กำหนดไวยากรณ์ใหม่

มันแก้ปัญหา 2 ข้อดังนี้:

  • polars_ta สามารถเขียนนิพจน์คำนวณฟีเจอร์ได้อย่างสะดวก แต่เมื่อเจอนิพจน์ที่ "ผสมผสาน Time Series กับ Cross Section" ด้วยการใช้ expr_codegen จะสามารถจัดกลุ่มอัตโนมัติช่วยประหยัดงานได้มาก
  • expr_codegen ใช้เทคนิค Common Subexpression Elimination เพื่อลดการคำนวณซ้ำซ้อนอย่างมากและเพิ่มประสิทธิภาพ

แม้แต่ในวงการ Quant นักวิจัยระดับต้นที่จำกัดแค่ตัวชี้วัด Time Series ใช้แค่ polars_ta ก็เพียงพอ ส่วนระดับกลางและสูงที่ใช้ตัวชี้วัด Cross Section แนะนำให้ใช้ expr_codegen

แม้โครงการนี้จะพึ่งพา polars_ta อย่างมากในตอนนี้ แต่ก็รองรับการแปลไปยังไลบรารีอื่น เช่น pandas / cudf.pandas เพียงแต่ตอนนี้ยังขาดไลบรารีที่ง่าย

หลักการทำงาน

โปรเจกต์นี้ใช้ไลบรารี sympy เป็นหลัก โดยใช้ฟังก์ชันสำคัญดังนี้:

  • simplify: ลดรูปนิพจน์ที่ซับซ้อน
  • cse: Common Subexpression Elimination กำจัดนิพจน์ย่อยที่ซ้ำกัน
  • StrPrinter: แสดงสตริงที่ต่างกันตามแต่ละฟังก์ชัน ปรับแต่งโค้ดนี้เพื่อรองรับภาษา/ไลบรารีอื่น

เนื่องจาก groupby, sort ใช้เวลามาก หากจัดกลุ่มสูตรล่วงหน้า แต่ละกลุ่มใช้ groupby ต่างกัน จะช่วยลดเวลาได้

  • ts_xxx(ts_xxx): คำนวณใน groupby เดียวกันได้
  • cs_xxx(cs_xxx): คำนวณใน groupby เดียวกันได้
  • ts_xxx(cs_xxx): ต้องคำนวณใน groupby ต่างกัน
  • cs_xxx(ts_xxx(cs_xxx)): ต้องคำนวณใน groupby สามกลุ่มต่างกัน
  • gp_xxx(aa, )+gp_xxx(bb, ): เพราะ aa,bb ต่างกัน ต้อง groupby สองกลุ่ม

ดังนั้น

  • ต้องมีฟังก์ชันที่ดึงประเภทนิพจน์ปัจจุบัน (get_current) และประเภทนิพจน์ย่อย (get_children)
  • หากประเภทปัจจุบันกับประเภทย่อยต่างกัน สามารถแยกสูตรสั้นออกมา (extract) นิพจน์ประเภทเดียวกันแต่คนละชั้นมีลำดับ ต้องไม่อยู่ใน groupby เดียวกัน
  • ใช้คุณสมบัติของ cse แทนที่นิพจน์ยาวด้วยนิพจน์สั้นที่แยกออกมา แล้วนำเข้าสู่กราฟเวียนทิศทาง (DAG)
  • ใช้การไหลของ DAG เพื่อแบ่งชั้น ชั้นเดียวกันของ ts, cs, gp ไม่ต้องแยกลำดับ
  • แต่ละชั้นจัดกลุ่ม ts, cs, gp แล้วสร้างโค้ด (codegen)

ข้อมูลแฝง

  • ts: sort(by=[ASSET, DATE]).groupby(by=[ASSET], maintain_order=True)
  • cs: sort(by=[DATE]).groupby(by=[DATE], maintain_order=False)
  • gp: sort(by=[DATE, GROUP]).groupby(by=[DATE, GROUP], maintain_order=False)

กล่าวคือ

  • ฟังก์ชัน Time Series ซ่อนฟิลด์ ASSET, DATE ฟังก์ชัน Cross Section ซ่อนฟิลด์ DATE
  • ฟังก์ชันกลุ่มจะเพิ่มฟิลด์ GROUP พร้อมซ่อนฟิลด์ DATE

วิธีจำแนกสองแบบ

  • จำแนกตามคำนำหน้าฟังก์ชัน (get_current_by_prefix) ต้องขึ้นต้นด้วย ts_, cs_, gp_
  • จำแนกตามชื่อเต็มของฟังก์ชัน (get_current_by_name) ไม่ต้องจำกัดชื่อ เช่น cs_rank อาจใช้ชื่อ rank

การจัดการ Null

null เกิดจากอะไร?

  • การหยุดซื้อขาย ถูกกรองออกก่อนคำนวณ ไม่กระทบการคำนวณถัดไป
  • เวลาซื้อขายแต่ละสินค้าต่างกัน
  • เกิดจากการคำนวณ null ที่ปลายชุดข้อมูลไม่กระทบผลลัพธ์ของฟังก์ชัน Time Series แต่ถ้า null อยู่กลางชุดข้อมูลจะมีผล เช่น if_else(close<2, None, close)

https://github.com/pola-rs/polars/issues/12925#issuecomment-2552764629

ไอเดียดีมาก สรุปได้ 2 วิธี:

  • แยก null เป็นกลุ่มหนึ่ง not_null อีกกลุ่ม ต้องเรียกสองรอบ
  • กลุ่มเดียว แต่เรียงลำดับซ้อนกัน null อยู่ข้างหน้า not_null อยู่ข้างหลัง เรียกครั้งเดียว เร็วกว่านิดหน่อย

X1 = (ts_returns(CLOSE, 3)).over(CLOSE.is_not_null(), _ASSET_, order_by=_DATE_),
X2 = (ts_returns(CLOSE, 3)).over(_ASSET_, order_by=[CLOSE.is_not_null(), _DATE_]),
X3 = (ts_returns(CLOSE, 3)).over(_ASSET_, order_by=_DATE_),

แบบที่ 2 พื้นที่ null ต้นชุดข้อมูลจะมีผลหรือไม่ขึ้นอยู่กับฟังก์ชัน โดยเฉพาะกับ input หลายคอลัมน์

  • over_null='partition_by' แยกเป็น 2 โซน
  • over_null='order_by' อยู่ในโซนเดียว null อยู่ข้างหน้า
  • over_null=None ไม่จัดการ null เลย เร็วกว่า ถ้ามั่นใจว่าไม่มี null กลางชุดข้อมูล แนะนำใช้พารามิเตอร์นี้

codegen_exec(over_null='partition_by') จะใช้ partition_by ทั่วไป แต่ถ้าเจอฟังก์ชันแบบ ts_count_nulls ต้องใช้ over_null=None ดังนั้นเครื่องมือนี้จึงเพิ่มคอมเมนต์เพื่อระบุพารามิเตอร์ในแต่ละบรรทัด

  • # --over_null partition_by ใช้ over_null='partition_by' แค่บรรทัดเดียว
  • # --over_null=order_by ใช้ over_null='order_by' แค่บรรทัดเดียว
  • # --over_null ใช้ over_null=None แค่บรรทัดเดียว
  • # ใช้ค่าจากพารามิเตอร์ over_null ที่ส่งเข้า codegen_exec

หมายเหตุ:

  • # --over_null ต้องเขียนต่อท้ายบรรทัดนิพจน์เท่านั้น เขียนแยกบรรทัดจะถูกละเลย
  • # --over_null # --over_null=order_by ถ้ามีหลาย # จะใช้แค่ตัวแรก
  • มีผลแค่กับฟังก์ชัน ts ที่อยู่นอกสุด ถ้า ts ไม่อยู่นอกสุดต้องแยกเอง เช่น
   X1 = cs_rank(ts_mean(CLOSE, 3)) # --over_null=order_by # จะไปใช้กับ cs_rank ซึ่งไม่ถูกต้อง
   X2 = ts_rank(ts_mean(CLOSE, 3), 5) # --over_null=order_by # คิดว่าจะใช้กับ ts_rank(ts_mean) แต่ถ้ามี ts_mean ซ้ำจริงๆ จะใช้กับ ts_rank(_x_0)
   ``
   ควรเขียนใหม่เป็น
   `python
   _x_0 = ts_mean(CLOSE, 3)  # --over_null=order_by 
   X1 = cs_rank(_x_0)
   X2 = ts_rank(_x_0, 5)
   `

ไวยากรณ์พิเศษ
  • รองรับนิพจน์สามทาง C?T:F (ใช้ได้ในสตริงเท่านั้น) ด้านล่างจะถูกแปลงเป็น C or True if( T )else F แล้วแก้เป็น T if C else F สุดท้ายเป็น if_else(C,T,F) รองรับใช้ร่วมกับ if else
    • 

  • (A-1 จะถูกแปลงเป็น int_(A-1
    • 

  • ป้องกัน A==B ถูก sympy แปลงเป็น False จะใช้ Eq(A,B)
    • 

  • ความหมายของ A^B ขึ้นอยู่กับพารามิเตอร์ convert_xor ถ้า convert_xor=True จะเป็น Pow(A,B) ถ้า False จะเป็น Xor(A,B) ค่า default คือ False ใช้  สำหรับยกกำลัง
    • 

  • รองรับ A&B&C แต่ไม่รองรับ A==B==C ถ้า C เป็น boolean, AB เป็นตัวเลข แทนด้วย (A==B)==C ถ้า ABC เป็นตัวเลขแทนด้วย (A==B)&(B==C)
    • 

  • ไม่รองรับ A<=B<=C ให้แทนด้วย (A<=B)&(B<=C)
    • 

  • รองรับ A[0]+B[1]+C[2] จะถูกแปลงเป็น A+ts_delay(B,1)+ts_delay(C,2)
    • 

  • รองรับ ~A จะถูกแปลงเป็น Not(A)
    • 

  • ฟังก์ชันที่ขึ้นต้นด้วย gp_ จะคืนค่าฟังก์ชัน cs_ ที่ตรงกัน เช่น gp_func(A,B,C) จะถูกแทนที่ด้วย cs_func(B,C) โดยใช้ A ใน groupby([date, A])
    • 

  • รองรับ tuple unpacking เช่น A,B,C=MACD() ด้านล่างจะแปลงเป็น
    •

   B = unpack(_x_0, 1)

   `
