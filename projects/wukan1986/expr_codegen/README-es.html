expr_codegen Traductor de Expresiones

Antecedentes del proyecto

Después de lanzar mi nueva biblioteca polars_ta, reflexioné sobre qué es realmente expr_codegen.

En esencia, expr_codegen es un DSL, un Lenguaje Específico de Dominio (Domain Specific Language). Pero no define una nueva sintaxis.

Resuelve dos problemas:

  • polars_ta ya permite escribir fácilmente expresiones para el cálculo de características, pero al encontrar expresiones que mezclan series temporales y transversales, el uso de expr_codegen permite agrupar automáticamente, ahorrando mucho trabajo.
  • expr_codegen utiliza la Eliminación de Subexpresiones Comunes (Common Subexpression Elimination), lo que reduce considerablemente los cálculos repetidos y mejora la eficiencia.

Incluso en el campo cuantitativo, los investigadores principiantes limitados a indicadores temporales pueden usar solo polars_ta, mientras que los investigadores intermedios y avanzados que usan indicadores transversales deberían usar expr_codegen.

Aunque actualmente este proyecto depende estrechamente de polars_ta, también es compatible para traducirse a otras bibliotecas como pandas / cudf.pandas, solo que por ahora falta una biblioteca sencilla para ello.

Principio de funcionamiento

Este proyecto depende del proyecto sympy. Las funciones principales utilizadas son:

  • simplify: simplifica expresiones complejas
  • cse: Eliminación de Subexpresiones Comunes (Common Subexpression Elimination)
  • StrPrinter: imprime diferentes cadenas según la función. Personalizando este código se pueden soportar otros lenguajes o bibliotecas

Dado que groupby y sort consumen mucho tiempo, si se clasifican las fórmulas por adelantado y diferentes categorías usan diferentes groupby, se puede reducir el tiempo de cómputo.

  • ts_xxx(ts_xxx): Puede calcularse en el mismo groupby
  • cs_xxx(cs_xxx): Puede calcularse en el mismo groupby
  • ts_xxx(cs_xxx): Debe calcularse en diferentes groupby
  • cs_xxx(ts_xxx(cs_xxx)): Debe calcularse en tres groupby diferentes
  • gp_xxx(aa, )+gp_xxx(bb, ): Debido a que aa y bb son diferentes, deben calcularse en dos groupby diferentes

Por lo tanto:

  • Se necesita una función para obtener la categoría de la expresión actual (get_current) y la categoría de las subexpresiones (get_children)
  • Si la categoría actual y la de los hijos son diferentes, se puede extraer una fórmula corta (extract). Expresiones del mismo tipo en diferentes niveles tienen relación de precedencia y no pueden colocarse en el mismo groupby
  • Aprovechando la característica de cse, las expresiones largas se reemplazan por las fórmulas cortas extraídas previamente. Luego se ingresan en un grafo acíclico dirigido (DAG)
  • Utilizando el flujo del grafo acíclico dirigido, se realiza la estratificación. En la misma capa, ts, cs, gp no se distinguen por orden
  • En la misma capa se agrupan ts, cs, gp, y luego se genera el código (codegen)

Información implícita:

  • ts: sort(by=[ASSET, DATE]).groupby(by=[ASSET], maintain_order=True)
  • cs: sort(by=[DATE]).groupby(by=[DATE], maintain_order=False)
  • gp: sort(by=[DATE, GROUP]).groupby(by=[DATE, GROUP], maintain_order=False)

Es decir:

  • Las funciones temporales ocultan dos campos: ASSET, DATE; las funciones transversales ocultan uno: DATE
  • Las funciones de agrupación agregan el campo GROUP y ocultan el campo DATE

Dos métodos de clasificación:

  • Clasificación por prefijo del operador (get_current_by_prefix), donde el operador debe comenzar con ts_, cs_, gp_
  • Clasificación por nombre completo del operador (get_current_by_name), sin restringir el nombre. Por ejemplo, cs_rank puede llamarse simplemente rank

Manejo de Null

¿Cómo se produce null?

  • Por suspensión de la cotización. Antes del cálculo se filtran directamente, no afectan los cálculos posteriores.
  • Diferentes variedades tienen diferentes horarios de negociación
  • Por el cálculo. null en los extremos de la serie no afecta los resultados de los operadores temporales, pero si aparece en medio sí afecta. Por ejemplo: if_else(close<2, None, close)

https://github.com/pola-rs/polars/issues/12925#issuecomment-2552764629

Excelente idea, en resumen hay dos formas de implementarlo:

  • Dividir null en un grupo y not_null en otro. Requiere dos llamadas.
  • Solo un grupo, pero con orden compuesto, colocando null al principio y not_null al final. Solo una llamada, un poco más rápido.

X1 = (ts_returns(CLOSE, 3)).over(CLOSE.is_not_null(), _ASSET_, order_by=_DATE_),
X2 = (ts_returns(CLOSE, 3)).over(_ASSET_, order_by=[CLOSE.is_not_null(), _DATE_]),
X3 = (ts_returns(CLOSE, 3)).over(_ASSET_, order_by=_DATE_),

En el caso 2, si la región de null al principio afecta el resultado depende del operador, especialmente cuando hay múltiples columnas de entrada, la región de null puede contener datos

  • over_null='partition_by': divide en dos regiones
  • over_null='order_by': una sola región, null al principio
  • over_null=None: no se maneja, se llama directamente, más rápido. Se recomienda si se sabe que no habrá null intermedios

codegen_exec(over_null='partition_by') utiliza globalmente partition_by. Pero para funciones como ts_count_nulls que operan sobre null, se debe usar over_null=None. Por eso, la herramienta permite especificar parámetros por línea con comentarios:

  • # --over_null partition_by: por línea, over_null='partition_by'
  • # --over_null=order_by: por línea, over_null='order_by'
  • # --over_null: por línea, over_null=None
  • #: toma el valor de over_null pasado a codegen_exec

Notas:

  • Los comentarios de parámetro # --over_null solo pueden ir al final de la línea de expresión, no en línea separada, de lo contrario serán ignorados
  • Si hay varios # --over_null # --over_null=order_by, solo se toma el primero válido
  • Solo es efectivo para la función ts del nivel más externo. Si la función ts no está en el exterior, debe extraerse manualmente. Por ejemplo:
   X1 = cs_rank(ts_mean(CLOSE, 3)) # --over_null=order_by # Se aplica a cs_rank, sin sentido
   X2 = ts_rank(ts_mean(CLOSE, 3), 5) # --over_null=order_by # Se piensa que se aplica a ts_rank(ts_mean), pero por la existencia de ts_mean común, en realidad es para ts_rank(_x_0)
   ``
   Debe escribirse como
   `python
   _x_0 = ts_mean(CLOSE, 3)  # --over_null=order_by 
   X1 = cs_rank(_x_0)
   X2 = ts_rank(_x_0, 5)
   `

  • Debido a la facilidad de error, se recomienda encarecidamente generar el output_file para verificar si el código generado es correcto.
Limitaciones de expr_codegen
  • DAG solo puede agregar columnas, no eliminarlas. Si se agrega una columna con el mismo nombre, se sobrescribe.
    • 

  • No soporta eliminar filas, pero se puede agregar una columna de marcas de eliminación y luego eliminar filas externamente. Eliminar filas afecta a todas las columnas, lo que no cumple con el principio de DAG
    • 

  • No soporta reesampleo, por la misma razón que no soporta eliminar filas. Debe hacerse externamente
    • 

  • Eliminar filas y reesampleo pueden usarse como divisores para dividir grandes bloques de código en varios DAG encadenados. Es complejo y difícil de entender, por lo que finalmente no se implementó
Sintaxis especial
  • Soporta la expresión ternaria C?T:F (solo en cadenas), que internamente se convierte primero en C or True if( T )else F, después en T if C else F, y finalmente en if_else(C,T,F). Soporta mezcla con if else
    • 

  • (A-1 se transforma internamente en int_(A-1
    • 

  • Para evitar que A==B sea reemplazado por False por sympy, se convierte en Eq(A,B)
    • 

  • El significado de A^B depende del parámetro convert_xor: si convert_xor=True, se convierte en Pow(A,B), de lo contrario en Xor(A,B). Por defecto es False, se usa  para potencia
    • 

  • Soporta A&B&C, pero no A==B==C. Si C es booleano y AB son valores, puede reemplazarse manualmente por (A==B)==C. Si ABC son valores, puede reemplazarse por (A==B)&(B==C)
    • 

  • No soporta A<=B<=C, debe reemplazarse manualmente por (A<=B)&(B<=C)
    • 

  • Soporta A[0]+B[1]+C[2], que se traduce en A+ts_delay(B,1)+ts_delay(C,2)
    • 

  • Soporta ~A, que se traduce en Not(A)
    • 

  • Las funciones que empiezan con gp_ devuelven la función cs_ correspondiente. Por ejemplo, gp_func(A,B,C) se reemplaza por cs_func(B,C), donde A se usa en groupby([date, A])
    • 

  • Soporta desempaquetado de tuplas A,B,C=MACD(), que internamente se reemplaza por:
    • 
    
`python
    _x_0 = MACD()
    A = unpack(_x_0, 0)
    B = unpack(_x_0, 1)
    C = unpack(_x_0, 2)
    `

  • Los comentarios en línea soportan entrada de parámetros, como: # --over_null, # --over_null=order_by, # --over_null=partition_by
    • 

  • En los bloques de código, las sentencias import, def y class se insertan automáticamente tal cual en el código generado
Variables con guion bajo inicial
  • En los datos de salida, todas las columnas que comienzan con _ se eliminan automáticamente. Así que las variables que se quieran conservar no deben comenzar con _
    • 

  • Para reducir cálculos repetidos, se agregan automáticamente variables intermedias que comienzan con _x_, como _x_0, _x_1, etc. Al final se eliminan automáticamente
    • 

  • Si una expresión de una sola línea es muy larga o hay cálculos repetidos, se puede dividir en varias líneas usando variables intermedias que comienzan con _. Estas recibirán automáticamente un sufijo numérico, formando variables diferentes como _A_0_, _A_1_, etc. Usos:
    • 
    
  • El mismo nombre de variable se usa varias veces, pero en realidad son variables diferentes
    • 
    
  • Asignación en bucles, pero DAG no soporta ciclos. Las variables con el mismo nombre a ambos lados del = son realmente diferentes
